Key Findings & Gaps

Critical Issues:

  • Incomplete Migration: Only ~30% of buttons use the shared Button component
  • Multiple Patterns: Three different implementation approaches running simultaneously
  • Missing Variants: 12+ button types have no equivalent in the Button component
  • Inconsistent Usage: Same functionality implemented differently across modules

Missing from Button Component:

  • Login-specific styling (full-width, larger size)
  • Refresh button variant
  • Review button (yellow/warning clickable)
  • QC button (clickable warning style)
  • Icon-only buttons
  • Certificate action buttons
  • Close button (×) pattern
  • Tab and navigation button variants
  • Status button with gradient styles
  • Transfer state buttons

Implementation Patterns:

  • Button Component: Used primarily in modals and forms
  • CSS Modules: Used in GaugeRow and similar list components
  • Legacy Classes: Still present in many older components
  • Inline Styles: Used for close buttons and some custom implementations
  • Interactive Elements: Cards and divs with onClick handlers

Migration Recommendations

Priority 1: Extend Button Component

  • Add fullWidth prop for login buttons
  • Add iconOnly prop for icon buttons
  • Add missing variants: refresh, review, certificate, status
  • Add gradient support for special status buttons
  • Create TabButton and NavButton sub-components

Priority 2: Standardize Implementation

  • Refactor GaugeRow to use Button component instead of CSS modules
  • Create migration guide with 1:1 mapping
  • Replace all legacy class usage with Button component
  • Standardize close button pattern

Priority 3: Component Library

  • Create Storybook documentation for all button variants
  • Add visual regression tests
  • Document migration patterns
  • Create lint rules to enforce Button component usage
