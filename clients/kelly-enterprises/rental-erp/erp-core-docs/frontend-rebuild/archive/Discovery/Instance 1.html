Frontend UI Comparison - Phase 1: Buttons

Analysis by Instance 1 - Comprehensive comparison of button implementations between Legacy and Modular systems

Executive Summary

This analysis compares button implementations across the Legacy and Modular frontend architectures. The investigation reveals significant differences in approach, styling methods, and component structure.

Key Findings

  • Legacy uses global CSS classes (.btn, .btn-primary) with direct className application
  • Modular uses a TypeScript React component with CSS Modules for scoped styling
  • Both systems support similar variant types but with different naming conventions
  • Modular system adds loading states and icon support not present in legacy
  • Legacy includes specialized buttons (btn-unseal) not yet in modular Button component

MODULAR Implementation

Component (Button.tsx)
interface ButtonProps extends ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: 'primary' | 'secondary' | 'success' | 'warning' | 'danger' | 'info' | 'default' | 'outline';
  size?: 'sm' | 'md' | 'lg';
  loading?: boolean;
  icon?: ReactNode;
  children: ReactNode;
}

export const Button = ({
  variant = 'primary',
  size = 'md',
  loading = false,
  icon,
  children,
  disabled,
  className = '',
  ...props
}: ButtonProps) => {
  const sizeClass = size === 'sm' ? styles.small : size === 'lg' ? styles.large : styles.medium;
  const variantClass = styles[variant] || styles.primary;
  const classes = `${styles.button} ${variantClass} ${sizeClass} ${className}`;

  return (
    <button className={classes} disabled={disabled || loading} {...props}>
      {loading ? <LoadingSpinner /> : icon && <span>{icon}</span>}
      {children && <span>{children}</span>}
    </button>
  );
};

Usage Patterns Discovered

Legacy System Patterns

  • Direct className usage: Components directly apply CSS classes
  • Custom action buttons: Many components define their own button styles (save-btn, cancel-btn, etc.)
  • Specialized variants: btn-unseal for specific gauge operations
  • Inline styles: Some components use inline button elements without consistent classes

Modular System Patterns

  • Component import: Import Button from infrastructure
  • Props-based configuration: Control appearance via props
  • Consistent API: All buttons use the same component interface
  • Composition: Icon and loading states handled internally

Hybrid Usage in Modular

Finding: Some modular components (like GaugeRow.tsx) still use direct button elements with custom classes instead of the Button component. This suggests incomplete migration.

Gaps and Recommendations

Missing in Modular Implementation

btn-unseal variant: The orange gradient "Unseal" button style from legacy is not available in the modular Button component.
Inconsistent adoption: Many components in the modular system still use custom button implementations instead of the shared Button component.

Implementation Differences

Size naming: Legacy uses 'btn-sm' and 'btn-lg' while modular uses 'sm', 'md', 'lg' props.
Default variant: Legacy has no default (must specify btn-primary, etc.) while modular defaults to 'primary'.

New Features in Modular

Loading states: Built-in loading spinner animation with automatic disabled state.
Icon support: Dedicated icon prop with automatic spacing and alignment.
Outline variant: New outline style not present in legacy system.
TypeScript support: Full type safety and IntelliSense support.

Migration Checklist

For Full Button Migration

  • Add 'unseal' variant to modular Button component
  • Replace all custom button implementations in GaugeRow and similar components
  • Standardize save-btn and cancel-btn patterns to use Button component
  • Ensure all modal action buttons use the shared component
  • Add any missing specialized button styles as variants
  • Document button usage patterns and guidelines
  • Create migration script to identify remaining legacy button usage
